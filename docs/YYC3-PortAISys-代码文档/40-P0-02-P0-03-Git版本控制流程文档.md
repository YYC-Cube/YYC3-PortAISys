## 二、分支管理策略

### 2.1 分支类型

#### 2.1.1 main分支

**用途**: 生产环境代码分支，始终保持稳定、可部署的状态

**特点**:
- 只接受来自develop分支和release分支的合并
- 每次合并都会打上版本标签
- 禁止直接推送代码到main分支
- 禁止直接在main分支上开发

**命名**: main

**保护规则**:
- 需要Pull Request才能合并
- 需要至少1个评审通过
- 需要通过所有CI检查
- 需要分支是最新的

#### 2.1.2 develop分支

**用途**: 开发环境代码分支，集成所有feature分支的最新代码

**特点**:
- 接受来自feature分支的合并
- 定期合并到release分支
- 可能包含未完成的功能
- 可能包含不稳定的代码

**命名**: develop

**保护规则**:
- 需要Pull Request才能合并
- 需要至少1个评审通过
- 需要通过所有CI检查

#### 2.1.3 feature分支

**用途**: 功能开发分支，用于开发新功能或修改现有功能

**特点**:
- 从develop分支创建
- 开发完成后合并回develop分支
- 可以存在多个feature分支
- 命名应清晰描述功能

**命名**: feature/<功能描述>

**示例**:
```
feature/transformer-optimization
feature/mixed-precision-training
feature/fault-detection-engine
```

**生命周期**:
1. 从develop分支创建
2. 开发功能
3. 提交代码
4. 创建Pull Request到develop
5. 代码评审
6. 合并到develop
7. 删除feature分支

#### 2.1.4 release分支

**用途**: 发布准备分支，用于准备新版本发布

**特点**:
- 从develop分支创建
- 只进行bug修复和版本号更新
- 测试完成后合并到main和develop分支
- 合并后打上版本标签

**命名**: release/<版本号>

**示例**:
```
release/v1.0.0
release/v1.1.0
```

**生命周期**:
1. 从develop分支创建
2. 修复bug
3. 更新版本号
4. 测试验证
5. 合并到main分支
6. 打上版本标签
7. 合并回develop分支
8. 删除release分支

#### 2.1.5 hotfix分支

**用途**: 紧急修复分支，用于修复生产环境的紧急bug

**特点**:
- 从main分支创建
- 修复完成后合并到main和develop分支
- 合并后打上版本标签
- 优先级最高

**命名**: hotfix/<版本号>

**示例**:
```
hotfix/v1.0.1
hotfix/v1.0.2
```

**生命周期**:
1. 从main分支创建
2. 修复bug
3. 更新版本号
4. 测试验证
5. 合并到main分支
6. 打上版本标签
7. 合并回develop分支
8. 删除hotfix分支

#### 2.1.6 bugfix分支

**用途**: bug修复分支，用于修复非紧急bug

**特点**:
- 从develop分支创建
- 修复完成后合并回develop分支
- 可以存在多个bugfix分支
- 命名应清晰描述bug

**命名**: bugfix/<bug描述>

**示例**:
```
bugfix/memory-leak
bugfix/database-connection-error
```

**生命周期**:
1. 从develop分支创建
2. 修复bug
3. 提交代码
4. 创建Pull Request到develop
5. 代码评审
6. 合并到develop
7. 删除bugfix分支

### 2.2 分支策略图

```

  ↑
  │ (merge)
  │

  ↑
  │ (merge)
  │

  ↑
  │ (merge)

```

## 三、代码提交规范

### 3.1 提交信息格式

#### 3.1.1 提交信息结构

```
<type>(<scope>): <subject>

<body>

<footer>
```

#### 3.1.2 Type（类型）

| Type | 描述 | 示例 |
|------|------|------|
| feat | 新功能 | feat(transformer): 添加多头注意力机制 |
| fix | 修复bug | fix(database): 修复数据库连接超时问题 |
| docs | 文档更新 | docs(readme): 更新安装说明 |
| style | 代码格式调整 | style(python): 调整代码缩进 |
| refactor | 代码重构 | refactor(api): 重构API接口 |
| perf | 性能优化 | perf(model): 优化模型推理速度 |
| test | 测试相关 | test(unit): 添加单元测试 |
| chore | 构建/工具相关 | chore(deps): 更新依赖库版本 |

#### 3.1.3 Scope（范围）

Scope用于说明提交影响的模块或功能，常见的Scope包括：

- transformer: Transformer优化模块
- training: 混合精度训练模块
- compression: 模型压缩模块
- distributed: 分布式训练模块
- detection: 故障检测引擎模块
- diagnosis: 故障诊断引擎模块
- recovery: 规则引擎恢复模块
- rl: 强化学习优化模块
- kg: 知识图谱模块
- monitoring: 监控告警模块
- api: API接口
- database: 数据库
- config: 配置文件
- docs: 文档
- tests: 测试

#### 3.1.4 Subject（主题）

Subject是对提交内容的简短描述，要求：

- 使用现在时态
- 首字母小写
- 结尾不加句号
- 不超过50个字符

**示例**:
```
feat(transformer): 添加多头注意力机制
fix(database): 修复数据库连接超时问题
docs(readme): 更新安装说明
```

#### 3.1.5 Body（正文）

Body是对提交内容的详细描述，要求：

- 使用现在时态
- 说明"是什么"和"为什么"，而不是"怎么做"
- 每行不超过72个字符
- 可以包含多个段落

**示例**:
```
feat(transformer): 添加多头注意力机制

添加多头注意力机制以提高模型性能。
多头注意力机制可以让模型同时关注不同位置的信息，
从而提高模型的表达能力。

主要改进：
- 支持8个注意力头
- 添加位置编码
- 优化计算效率
```

#### 3.1.6 Footer（脚注）

Footer用于说明Breaking Changes或关闭Issue。

**Breaking Changes**:
```
feat(api): 移除旧的API接口

移除旧的API接口以简化代码结构。
Breaking Change: 旧的API接口已移除，请使用新的API接口。
```

**关闭Issue**:
```
fix(bug): 修复登录问题

修复用户登录时的认证错误问题。
Closes #123
```

### 3.2 提交信息示例

#### 3.2.1 新功能提交

```
feat(transformer): 添加Flash Attention支持

添加Flash Attention以提高注意力机制的计算效率。
Flash Attention通过优化内存访问模式，
显著减少了注意力计算的时间和内存消耗。

主要改进：
- 支持Flash Attention 2
- 优化GPU内存使用
- 提高计算速度约2倍
```

#### 3.2.2 Bug修复提交

```
fix(database): 修复数据库连接池泄漏问题

修复数据库连接池未正确释放连接的问题。
该问题导致长时间运行后连接池耗尽，
影响系统稳定性。

修复内容：
- 确保连接在使用后正确释放
- 添加连接池监控
- 优化连接池配置

Closes #456
```

#### 3.2.3 文档更新提交

```
docs(api): 更新API文档

更新API文档以反映最新的接口变更。
添加了新的接口说明和示例代码。

更新内容：
- 添加故障检测API文档
- 更新模型训练API示例
- 添加错误码说明
```

#### 3.2.4 性能优化提交

```
perf(model): 优化模型推理速度

通过模型量化和算子融合优化模型推理速度。
优化后推理速度提升约30%，同时保持模型精度。

优化措施：
- 使用INT8量化
- 融合卷积和BN层
- 优化内存访问模式
```

### 3.3 提交频率

#### 3.3.1 提交原则

- 频繁提交：每天至少提交一次
- 小步提交：每次提交只做一件事
- 及时提交：完成一个功能点就提交
- 避免大提交：避免一次提交大量代码

#### 3.3.2 提交时机

- 完成一个功能点
- 修复一个bug
- 完成一个测试
- 更新文档
- 重构代码

#### 3.3.3 提交前检查

提交代码前，请确保：

- [ ] 代码通过本地测试
- [ ] 代码通过代码检查（lint）
- [ ] 代码格式符合规范
- [ ] 提交信息符合规范
- [ ] 没有敏感信息（密码、密钥等）
- [ ] 没有调试代码
- [ ] 没有注释掉的代码

### 3.4 提交命令

#### 3.4.1 基本提交

```bash
# 查看修改状态
git status

# 查看修改内容
git diff

# 添加修改到暂存区
git add <file>

# 提交修改
git commit -m "feat(transformer): 添加多头注意力机制"

# 推送到远程仓库
git push origin feature/transformer-optimization
```

#### 3.4.2 修改提交

```bash
# 修改最后一次提交
git commit --amend

# 修改最后一次提交信息
git commit --amend -m "fix(database): 修复数据库连接超时问题"

# 修改最后一次提交并添加文件
git add <file>
git commit --amend
```

#### 3.4.3 撤销提交

```bash
# 撤销最后一次提交（保留修改）
git reset --soft HEAD~1

# 撤销最后一次提交（不保留修改）
git reset --hard HEAD~1

# 撤销远程提交
git reset --hard HEAD~1
git push origin feature/xxx --force
```

## 四、代码合并流程

### 4.1 Pull Request流程

#### 4.1.1 创建Pull Request

**步骤**:
1. 在GitHub/GitLab上创建Pull Request
2. 选择目标分支（develop或main）
3. 填写PR标题和描述
4. 关联相关Issue
5. 添加评审人员
6. 提交Pull Request

**PR模板**:

```markdown
## 变更类型
- [ ] 新功能
- [ ] Bug修复
- [ ] 性能优化
- [ ] 代码重构
- [ ] 文档更新
- [ ] 测试相关
- [ ] 其他

## 变更描述
<!-- 简要描述本次PR的主要变更内容 -->

## 相关Issue
<!-- 关联相关的Issue，如：Closes #123 -->

## 测试情况
- [ ] 已添加单元测试
- [ ] 已通过本地测试
- [ ] 已通过CI测试

## 检查清单
- [ ] 代码符合项目编码规范
- [ ] 代码通过代码检查（lint）
- [ ] 代码格式符合规范
- [ ] 提交信息符合规范
- [ ] 没有敏感信息
- [ ] 没有调试代码
- [ ] 文档已更新

## 截图/演示
<!-- 如果有UI变更，请提供截图或演示链接 -->

## 备注
<!-- 其他需要说明的内容 -->
```

#### 4.1.2 PR标题规范

PR标题应遵循提交信息格式：

```
<type>(<scope>): <subject>
```

**示例**:
```
feat(transformer): 添加多头注意力机制
fix(database): 修复数据库连接超时问题
docs(api): 更新API文档
```

#### 4.1.3 PR评审流程

**评审步骤**:
1. 开发人员创建Pull Request
2. 系统自动运行CI检查
3. 评审人员评审代码
4. 评审人员提出修改意见
5. 开发人员根据意见修改代码
6. 评审人员再次评审
7. 评审通过后合并代码

**评审标准**:
- 代码质量：代码清晰、易读、易维护
- 功能正确：功能实现符合需求
- 性能优化：代码性能满足要求
- 安全性：没有安全漏洞
- 测试覆盖：有足够的测试用例
- 文档完整：有必要的文档和注释

**评审意见类型**:
- 必须修改：严重问题，必须修改后才能合并
- 建议修改：一般问题，建议修改但不强制
- 优化建议：代码优化建议，可以后续优化

#### 4.1.4 PR合并策略

**合并方式**:
- **Merge commit**: 保留完整的提交历史，推荐用于feature分支
- **Squash and merge**: 将多个提交合并为一个，推荐用于bugfix分支
- **Rebase and merge**: 保持线性提交历史，不推荐使用

**合并条件**:
- [ ] 至少1个评审通过
- [ ] 所有CI检查通过
- [ ] 没有未解决的评审意见
- [ ] 没有合并冲突

## 五、冲突解决机制

### 5.1 冲突预防

#### 5.1.1 预防措施

- 频繁同步：每天至少同步一次develop分支
- 小步提交：避免长时间不提交代码
- 及时合并：及时合并feature分支到develop
- 沟通协调：多人修改同一文件时提前沟通
- 代码分离：尽量减少多人修改同一文件

#### 5.1.2 同步develop分支

```bash
# 切换到feature分支
git checkout feature/transformer-optimization

# 拉取最新代码
git pull origin feature/transformer-optimization

# 合并develop分支
git fetch origin
git merge origin/develop

# 如果有冲突，解决冲突后提交
git add .
git commit -m "chore: 合并develop分支"
git push origin feature/transformer-optimization
```

### 5.2 冲突检测

#### 5.2.1 查看冲突

```bash
# 查看冲突文件
git status

# 查看冲突内容
git diff

# 查看冲突标记
<<<<<<< HEAD
当前分支的代码
=======
合并分支的代码
>>>>>>> develop
```

#### 5.2.2 冲突类型

- **内容冲突**: 同一行代码被不同分支修改
- **重命名冲突**: 文件被重命名或删除
- **二进制冲突**: 二进制文件被不同分支修改

### 5.3 冲突解决

#### 5.3.1 解决内容冲突

**步骤**:
1. 打开冲突文件
2. 查看冲突标记
3. 选择保留的代码或手动合并
4. 删除冲突标记
5. 保存文件
6. 添加文件到暂存区
7. 提交修改

**示例**:
```python
<<<<<<< HEAD
def calculate_attention(self, query, key, value):
    # 当前分支的实现
    return attention_scores
=======
def calculate_attention(self, query, key, value, mask=None):
    # 合并分支的实现
    if mask is not None:
        attention_scores = attention_scores.masked_fill(mask == 0, -1e9)
    return attention_scores
>>>>>>> develop
```

**解决后**:
```python
def calculate_attention(self, query, key, value, mask=None):
    # 合并后的实现
    attention_scores = self._calculate_scores(query, key, value)
    if mask is not None:
        attention_scores = attention_scores.masked_fill(mask == 0, -1e9)
    return attention_scores
```

#### 5.3.2 解决重命名冲突

```bash
# 查看重命名冲突
git status

# 选择保留的文件
git add <保留的文件>
git rm <删除的文件>

# 提交修改
git commit -m "chore: 解决重命名冲突"
```

### 6.2 版本标签

#### 6.2.1 版本号规范

采用语义化版本（Semantic Versioning）：

```
<major>.<minor>.<patch>
```

- **major**: 不兼容的API变更
- **minor**: 向下兼容的功能性新增
- **patch**: 向下兼容的问题修正

**示例**:
- v1.0.0: 初始版本
- v1.0.1: 修复bug
- v1.1.0: 添加新功能
- v2.0.0: 重大更新

#### 6.2.2 创建版本标签

```bash
# 创建轻量标签
git tag v1.0.0

# 创建附注标签（推荐）
git tag -a v1.0.0 -m "Release version 1.0.0"

# 推送标签到远程仓库
git push origin v1.0.0

# 推送所有标签
git push origin --tags
```

#### 6.2.3 查看版本标签

```bash
# 查看所有标签
git tag

# 查看标签详情
git show v1.0.0

# 查看匹配的标签
git tag -l "v1.*"
```

#### 6.2.4 删除版本标签

```bash
# 删除本地标签
git tag -d v1.0.0

# 删除远程标签
git push origin --delete v1.0.0
```

### 6.4 Git Hooks

#!/bin/bash

# 运行代码检查

    echo "代码检查失败，请修复后再提交"
    exit 1

#### 6.4.2 Commit-msg Hook

**.git/hooks/commit-msg**:
```bash
#!/bin/bash

# 检查提交信息格式
commit_regex='^(feat|fix|docs|style|refactor|perf|test|chore)(\(.+\))?: .{1,50}'

if ! grep -qE "$commit_regex" "$1"; then
    echo "提交信息格式不正确，请使用以下格式："
    echo "<type>(<scope>): <subject>"
    echo ""
    echo "Type: feat, fix, docs, style, refactor, perf, test, chore"
    echo "Scope: transformer, training, compression, etc."
    echo "Subject: 简短描述（不超过50个字符）"
    exit 1
fi

exit 0
```
